## 질문사항 답변

### 1. 이 프로젝트에서 AI가 UI 생성 후 버튼 클릭 이벤트는 어떻게 잡는가?

이 프로젝트는 "AI가 직접 프론트엔드 이벤트 리스너 코드를 생성"하는 방식이 아니라, spec 안의 `on.press` 액션 선언을 런타임이 해석해서 처리한다.

- 컴포넌트 구현은 `emit("press")` 같은 이벤트만 올린다. 예: 버튼 컴포넌트 예시는 [`README.md`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/README.md) 에서 `onClick={() => emit("press")}` 형태다.
- 실제 이벤트 해석은 React 렌더러의 [`packages/react/src/renderer.tsx`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/packages/react/src/renderer.tsx) 에서 한다.
  - 각 element의 `on` 필드를 읽고
  - `emit("press")` 호출 시 해당 이벤트에 연결된 액션 바인딩을 찾아
  - `execute(...)` 로 순차 실행한다.
- 액션 실행 로직은 [`packages/react/src/contexts/actions.tsx`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/packages/react/src/contexts/actions.tsx) 에 있다.
  - 내장 액션: `setState`, `pushState`, `removeState`, `push`, `pop`, `validateForm`
  - 커스텀 액션도 handler 등록 방식으로 처리 가능

즉, 흐름은 아래와 같다.

1. AI가 spec에 `on.press` 를 넣는다.
2. 렌더러가 element의 `on` 정의를 읽는다.
3. 실제 UI 컴포넌트가 `emit("press")` 를 호출한다.
4. `ActionProvider` 가 연결된 액션을 실행하고 state를 바꾼다.
5. state가 바뀌면 `visible`, `$state`, `repeat` 같은 바인딩이 다시 평가되어 UI가 갱신된다.

예를 들면 agent 프롬프트도 버튼 클릭 예시를 직접 강제한다. [`examples/chat/lib/agent.ts`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/examples/chat/lib/agent.ts)

```json
{
  "on": {
    "press": {
      "action": "setState",
      "params": {
        "statePath": "/submitted",
        "value": true
      }
    }
  }
}
```

정리하면, 이벤트는 "컴포넌트 -> emit -> renderer/action provider -> state 변경" 구조로 잡는다.

### 2. AI가 생성한 UI가 틀린 경우 어떻게 되는가?

틀린 경우는 종류별로 대응 방식이 다르다.

#### 2-1. 구조적으로 잘못된 spec인 경우

`@json-render/core` 에 [`packages/core/src/spec-validator.ts`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/packages/core/src/spec-validator.ts) 가 있어서 다음 같은 오류를 검사한다.

- `root` 누락
- `root` 가 `elements` 에 없음
- 없는 child 참조
- `visible`, `on`, `repeat`, `watch` 를 `props` 안에 잘못 넣은 경우

일부는 `autoFixSpec(...)` 으로 자동 보정도 가능하다. 예를 들어 `props.on` 을 top-level `on` 으로 옮기는 식이다.

#### 2-2. 없는 컴포넌트를 AI가 쓴 경우

React 렌더러는 registry에 없는 타입이면 `fallback` 을 사용한다. 예제 chat 앱은 [`examples/chat/lib/render/renderer.tsx`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/examples/chat/lib/render/renderer.tsx) 에서 fallback을 연결해둔다.

즉, 전체 화면이 죽기보다 "이 타입은 렌더링할 수 없음" 같은 대체 표시로 흡수하는 구조다.

#### 2-3. child 참조가 틀리거나 일부 element가 비정상인 경우

[`packages/react/src/renderer.tsx`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/packages/react/src/renderer.tsx) 에서 child가 없으면 warning만 남기고 그 element는 렌더하지 않는다.

즉, 부분적으로 빠질 수는 있지만 전체 렌더를 가능한 유지하려고 설계되어 있다.

#### 2-4. 특정 컴포넌트 렌더 중 런타임 에러가 나는 경우

같은 파일의 `ElementErrorBoundary` 가 개별 element 렌더 오류를 잡는다. 이 경우 해당 element만 사라지고 전체 앱은 계속 살아있다.

즉, "한 컴포넌트 실패 = 전체 화면 크래시"가 아니라 "문제 element 격리" 쪽이다.

#### 2-5. 의미적으로 틀린 UI인 경우

예를 들어 사용자가 원한 화면과 다르게 배치했거나, 정보 구조가 부정확한 경우는 validator가 자동으로 잡아주지 못한다. 이 경우 이 프로젝트의 기본 전략은 다음 두 가지다.

- catalog/prompt로 AI가 쓸 수 있는 컴포넌트와 액션을 강하게 제한
- state 기반 데이터 바인딩만 허용해서 hallucination을 줄임

특히 chat 예제 agent는 [`examples/chat/lib/agent.ts`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/examples/chat/lib/agent.ts) 에서 "항상 tool 먼저 호출", "실데이터를 `/state` 에 넣고 참조", "없는 데이터는 만들지 말 것"을 명시한다.

결론적으로, 틀린 UI가 나오면:

- 구조 오류는 검증/자동보정 가능
- 알 수 없는 컴포넌트는 fallback 처리
- 일부 child 오류는 부분 렌더
- 개별 컴포넌트 에러는 error boundary로 격리
- 의미적 오답은 프롬프트/카탈로그 제약으로 최대한 줄이되, 완전 자동 보정되지는 않는다

### 3. UI generation flow가 궁금하다. agent가 생각하는 단계 같은 게 정리되었으면 한다.

현재 저장소 기준으로 가장 잘 정리된 흐름은 chat 예제 agent다. 핵심 파일은 [`examples/chat/lib/agent.ts`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/examples/chat/lib/agent.ts), [`examples/chat/app/api/generate/route.ts`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/examples/chat/app/api/generate/route.ts), [`docs/chat-agent.md`](/Users/dodo/workspace/projects/a2ui-chat-poc/references/json-render/docs/chat-agent.md) 이다.

#### 전체 플로우

1. 사용자가 질문한다.
2. 서버 route가 message를 받아 agent에 전달한다.
3. agent는 먼저 tool 호출로 필요한 데이터를 수집한다.
4. 수집한 데이터를 짧게 요약한다.
5. 이어서 ````spec` fence 안에 JSONL patch(spec stream)를 출력한다.
6. 서버는 `pipeJsonRender(...)` 로 일반 텍스트와 spec patch를 분리해서 스트리밍한다.
7. 클라이언트는 patch를 누적 적용하면서 UI를 점진적으로 렌더링한다.
8. 렌더된 UI는 이후 버튼 클릭 등으로 state를 바꾸며 상호작용한다.

#### agent의 "생각 단계"를 실무적으로 풀면

완전한 chain-of-thought가 코드에 노출되어 있지는 않지만, 시스템 프롬프트 기준 의도된 단계는 아래처럼 정리할 수 있다.

1. 요청 분류
   - 날씨, GitHub, 코인, HN, 일반 검색 중 어떤 tool이 필요한지 판단
2. 데이터 수집
   - 적절한 tool을 먼저 호출
   - 필요한 경우 여러 tool을 조합
3. 상태 모델 설계
   - 가져온 데이터를 `/state/...` 어디에 둘지 결정
   - 차트용 배열, 표용 배열, metric용 scalar를 state path로 정리
4. UI 구조 설계
   - `Card`, `Grid`, `Metric`, `Table`, `Chart`, `Accordion` 등 중 적합한 컴포넌트 선택
   - 카탈로그 제약을 지키며 element tree 구성
5. 인터랙션 설계
   - 필요한 경우 `visible`, `repeat`, `on.press`, `$cond` 추가
   - 입력 컴포넌트는 `$bindState` 또는 액션으로 state와 연결
6. 스트리밍 출력
   - 요약 텍스트 출력
   - 이어서 JSONL patch를 순서대로 출력
   - `/state` 와 `/elements` 를 섞어 내보내며 점진 렌더링

#### 이 프로젝트에서 중요한 포인트

- agent는 "HTML을 생성"하는 게 아니라 "catalog에 허용된 컴포넌트로 된 spec"을 생성한다.
- 렌더러는 그 spec을 실제 React 컴포넌트로 바꾼다.
- 인터랙션도 임의 JS 생성이 아니라 `on.press -> action -> state` 방식이다.
- 따라서 이 프로젝트의 본질은 "LLM 자유생성 UI"보다는 "제약된 선언형 UI 생성기"에 가깝다.

#### 한 줄 요약

- 생성: agent가 tool로 데이터 수집 후 spec 생성
- 렌더: renderer가 spec을 컴포넌트 트리로 해석
- 상호작용: 액션이 state를 바꾸고 UI가 다시 계산됨
- 안정성: catalog, validator, fallback, error boundary로 방어
